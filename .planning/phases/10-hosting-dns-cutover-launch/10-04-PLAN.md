**Pre-condition:** All Plans 10-01 through 10-03 must be complete before executing this task.
**Timing:** Execute on a Tuesday–Thursday for the T-0 cutover. T-24h = the day before.

1. Verify current TTL values (should match what was recorded in DNS-RECORDS.md):
   ```bash
   dig +noall +answer proactivsports.com
   dig +noall +answer hk.proactivsports.com
   dig +noall +answer sg.proactivsports.com
   ```

2. Log into current DNS provider and lower the following record TTLs to 300 seconds (5 minutes):
   - A record @ (apex) → 300s
   - CNAME hk → 300s
   - CNAME sg → 300s
   - MX records → 300s (allows email continuity monitoring during cutover)

3. Wait for the original TTL to expire before proceeding (e.g. if original TTL was 3600s, wait 1 hour after the change before T-0). This ensures all resolvers see the 300s TTL before the NS change.

4. Confirm TTL change propagated:
   ```bash
   # Run these repeatedly until TTL shows 300 (or near it)
   dig +noall +answer +time=5 @8.8.8.8 proactivsports.com
   dig +noall +answer +time=5 @1.1.1.1 proactivsports.com
   ```

5. Update DNS-RECORDS.md "Pre-Cutover TTL State" table:
   - Record "Lowered at" timestamp for each record

ROLLBACK READINESS: With 300s TTL active, any DNS change during or after cutover propagates within 5 minutes. Rollback by reverting A/CNAME records to original values.

5. Begin monitoring propagation:
   ```bash

   dig +noall +answer NS proactivsports.com @8.8.8.8
   ```

HUMAN-ACTION — monitor after NS propagation confirms DNS resolves to Vercel.

1. In Vercel Dashboard → Project → Settings → Domains, monitor all three domains:
   - `proactivsports.com` — wait for status to change from "Pending Verification" to "Valid Configuration"
   - `hk.proactivsports.com` — same
   - `sg.proactivsports.com` — same

   Certificate provisioning typically completes within 5–15 minutes after DNS resolves to Vercel. Vercel uses ACME HTTP-01 challenge — it must be able to reach `/.well-known/acme-challenge/` at each domain, which now resolves to Vercel.

2. Verify SSL with curl after all three show "Valid Configuration":
   ```bash
   curl -sI https://proactivsports.com/ | head -1
   # Expected: HTTP/2 200 (or 3xx redirect — not SSL error)

   curl -sI https://hk.proactivsports.com/ | head -1
   # Expected: HTTP/2 200

   curl -sI https://sg.proactivsports.com/ | head -1
   # Expected: HTTP/2 200

   # Verify cert validity
   openssl s_client -connect proactivsports.com:443 -servername proactivsports.com </dev/null 2>&1 | grep "Verify return code"
   # Expected: Verify return code: 0 (ok)
   ```

3. If any domain fails cert provisioning:
   - Check `dig CNAME hk.proactivsports.com` resolves to the Vercel CNAME hash
   - Check A record: `dig +short proactivsports.com` returns Vercel IP
   - If DNS is wrong: fix in Cloudflare (propagates in ~5 minutes with 300s TTL)
   - If DNS is correct but cert still fails after 30 minutes: check Vercel Dashboard for error messages; try removing and re-adding the domain in Vercel

4. Do NOT proceed to Plan 10-05 until all three domains show "Valid Configuration" and SSL is verified.
