# Plan 09-04: vercel.json Redirect Rules and Verification

## Context

**Wave 3 — depends on Plan 09-03 T01–T04 being complete.** The `REDIRECT-MAP.json` must be fully populated with real crawl data before this plan runs. The HUMAN-ACTION steps in 09-03 (GSB check + crawl + traffic/backlink classification) must be signed off.

This plan takes the `redirects[]` array from `REDIRECT-MAP.json` and transcribes it into `vercel.json`. The transcription is straightforward: each `RedirectEntry` maps to a vercel.json redirect object with only `source`, `destination`, and `statusCode`. The extra metadata fields (`source_domain`, `target_market`, `has_backlinks`, `has_traffic`, `note`) live only in the JSON source and are not carried into `vercel.json`.

After committing `vercel.json`, the verify-redirects script is run against the new preview deployment to confirm every rule fires correctly.

**D-01:** Redirects in `vercel.json` fire BEFORE middleware — the legacy `.net` domains are not attached to Vercel yet, so the `Host: proactivsports.net` simulation via curl/fetch is the only way to test without DNS being live.

**D-02:** Every entry uses `"statusCode": 301` — never `"permanent": true` (which emits 308).

**Route budget check:** Before committing, verify the total Vercel route count is under 2,048. For a site this size (< 200 redirects), this is not a real constraint but must be confirmed.

## T01 — Transcribe REDIRECT-MAP.json redirects into vercel.json

**File:** `vercel.json`

**Current state:**
```json
{
  "$schema": "https://openapi.vercel.sh/vercel.json",
  "framework": "nextjs",
  "buildCommand": "pnpm build",
  "installCommand": "pnpm install --frozen-lockfile"
}
```

**Target state** (after adding redirects array):
```json
{
  "$schema": "https://openapi.vercel.sh/vercel.json",
  "framework": "nextjs",
  "buildCommand": "pnpm build",
  "installCommand": "pnpm install --frozen-lockfile",
  "redirects": [
    {
      "source": "/gymnastics",
      "destination": "https://hk.proactivsports.com/gymnastics/",
      "statusCode": 301
    }
  ]
}
```

**Transcription rules (binding):**
- Copy only `source`, `destination`, `statusCode` from each `RedirectEntry` in `REDIRECT-MAP.json`
- Do NOT carry `note`, `source_domain`, `target_market`, `has_backlinks`, `has_traffic` — vercel.json doesn't use these
- `statusCode` must be the integer `301` — not the string `"301"`, not `308`
- No `"permanent": true` — this emits 308, not 301 (D-02, Pitfall 2)
- No wildcard catch-all `"source": "/*"` — URL-by-URL only (D-12)
- All `destination` values must start with `https://proactivsports.com/`, `https://hk.proactivsports.com/`, or `https://sg.proactivsports.com/` (D-13)
- Include ALL entries from `REDIRECT-MAP.json` `redirects[]` array — entries with wildcard `:param*` patterns ARE included in vercel.json (Vercel supports them natively)

**Step-by-step:**
1. Open `.planning/phases/09-legacy-net-migration-security/REDIRECT-MAP.json`
2. For each entry in `redirects[]`, create the corresponding vercel.json entry
3. Add the entries to the `vercel.json` `redirects` array in the same order as the JSON source
4. Validate the JSON: `node -e "JSON.parse(require('fs').readFileSync('vercel.json', 'utf-8')); console.log('valid')"`
</task>

## T03 — HUMAN-ACTION: Run verify-redirects.mjs against preview

**This task requires the developer to obtain the current Vercel preview URL.**

**Steps:**

1. Commit `vercel.json` and push to a PR branch to trigger a Vercel preview deployment.

2. Get the preview URL from Vercel dashboard or the PR comment (format: `https://proactive-<sha>.vercel.app`).

3. Run the verification script:
   ```bash
   PREVIEW_URL=https://proactive-<sha>.vercel.app \
   node scripts/verify-redirects.mjs
   ```

4. If Vercel Authentication (preview protection from Phase 0) blocks the requests:
   ```bash
   # Get bypass token from Vercel dashboard → Project → Settings → Deployment Protection
   PREVIEW_URL=https://proactive-<sha>.vercel.app \
   VERCEL_BYPASS_TOKEN=<token-from-vercel> \
   node scripts/verify-redirects.mjs
   ```

5. Expected output:
   ```
   Verifying N redirect(s) against https://proactive-abc123.vercel.app

     PASS  /gymnastics → https://hk.proactivsports.com/gymnastics/
     PASS  /holiday-camps → https://hk.proactivsports.com/holiday-camps/
     ...
     SKIP  /blog/:slug* → (wildcard pattern — manual test required)

   Results: N passed, 0 failed
   ```

6. For wildcard patterns (e.g., `/blog/:slug*`), test manually:
   ```bash
   curl -s -o /dev/null -w "HTTP %{http_code}\nLocation: %{redirect_url}\n" \
     -H "Host: proactivsports.net" \
     "${PREVIEW_URL}/blog/some-blog-post-slug"
   # Expected: HTTP 301
   # Location: https://hk.proactivsports.com/blog/some-blog-post-slug
   ```

**Checkpoint:** All non-wildcard rules must show PASS before Phase 9 is considered complete for MIG-02.
</task>
