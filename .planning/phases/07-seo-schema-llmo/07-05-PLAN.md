## T1: Add correct sizes attribute to hero Image components on all homepages and location pages

<read_first>
- `app/root/page.tsx` — find hero `<Image>` component, check if `sizes` prop is present
- `app/hk/page.tsx` — find hero `<Image>` or video thumbnail, check `sizes` and `priority` props
- `app/sg/page.tsx` — find hero `<Image>`, check `sizes` prop
- `app/hk/wan-chai/page.tsx` — find hero image, check `sizes`
- `app/hk/cyberport/page.tsx` — find hero image, check `sizes`
- `app/sg/katong-point/page.tsx` — find hero image, check `sizes`
</read_first>

<action>
The research confirms: "Missing `sizes` prop is the primary LCP blocker." Without a correct `sizes` attribute, the browser downloads a much larger image than needed for the viewport, causing LCP to exceed 2.5s on mobile.

For each page listed in `files_modified`, find the above-fold hero `<Image>` component and add or update the `sizes` prop.

**Standard hero sizes pattern** (from UI-SPEC §8.2):
```tsx
sizes="(max-width: 768px) 100vw, (max-width: 1200px) 50vw, 1200px"
```

However, for full-width hero sections (where the image spans 100% of the viewport at all breakpoints), use:
```tsx
sizes="100vw"
```

Determine which pattern applies by checking the current CSS/Tailwind classes on the hero container:
- `w-full` or full-width heroes → `sizes="100vw"`
- Split-layout heroes (image + text side by side) → `sizes="(max-width: 768px) 100vw, (max-width: 1200px) 50vw, 1200px"`

Also verify `priority` is present on the hero image. The hero image MUST have `priority` to trigger preloading.

**What NOT to do:**
- Do NOT add `priority` to any non-hero image (below the fold images with `priority` block LCP preload)
- Do NOT change the `src` or `alt` of any existing images — only add/fix the `sizes` attribute

Apply this to:
- `app/root/page.tsx` — root homepage hero
- `app/hk/page.tsx` — HK homepage hero (may be a poster image for the Mux video)
- `app/sg/page.tsx` — SG homepage hero
- `app/hk/wan-chai/page.tsx` — Wan Chai location hero image
- `app/hk/cyberport/page.tsx` — Cyberport location hero image
- `app/sg/katong-point/page.tsx` — Katong Point location hero image
- `app/hk/gymnastics/page.tsx` — gymnastics pillar hero if present
- `app/sg/weekly-classes/page.tsx` — weekly classes hero if present
- `app/sg/prodigy-camps/page.tsx` — prodigy camps hero if present
</action>

<acceptance_criteria>
- `grep -r "sizes=" app/root/page.tsx app/hk/page.tsx app/sg/page.tsx` returns `sizes` prop on hero images
- `grep -r "sizes=" app/hk/wan-chai/page.tsx app/hk/cyberport/page.tsx app/sg/katong-point/page.tsx` returns `sizes` prop
- `grep -r "priority" app/root/page.tsx app/hk/page.tsx app/sg/page.tsx` returns at most 1 occurrence per page (hero only)
- `pnpm build` succeeds with no TypeScript errors
</acceptance_criteria>
</task>


For each `<Image priority>` found, determine:
1. Is this the FIRST visible image above the fold (hero image)? → KEEP `priority`
2. Is this below the fold, in a card, testimonial, logo wall, coach portrait, or any section below the hero? → REMOVE `priority`

Rules:
- ONLY the hero image (the first large image a user sees above the fold) should have `priority`
- Partner/sponsor logos → no `priority`
- Coach portrait thumbnails → no `priority`
- Programme card images → no `priority`
- Social proof/testimonial photos → no `priority`

After removing erroneous `priority` props, also verify each non-hero `<Image>` has appropriate `loading="lazy"` behavior (this is the Next.js `<Image>` default when `priority` is absent — no change needed, just confirming it's not overridden).

If a page has multiple `priority` props on images below the fold, remove them. If a page correctly has only one `priority` on the hero, no change needed.

## T3: Add Suspense boundary around BookingForm on all book-a-trial routes

<read_first>
- `app/hk/book-a-trial/page.tsx` — read full file to find the BookingForm client component
- `app/hk/book-a-trial/free-assessment/page.tsx` — read full file
- `app/sg/book-a-trial/page.tsx` — read full file
- Find the BookingForm component file (likely `components/hk/booking-form.tsx` or similar) — read it to confirm it's a client component
</read_first>

<action>
The research and UI-SPEC §8.2 identify: "Add `<Suspense fallback={<FormSkeleton />}>` around `BookingForm` client component on all book-a-trial routes."

This improves INP (Interaction to Next Paint) by allowing the page to render its static content while the booking form client bundle loads asynchronously.

**For each book-a-trial page that renders a `BookingForm` component:**

1. Create or locate a simple `FormSkeleton` component (or use an inline loading state). If no skeleton exists, create a minimal inline one:

```tsx
function FormSkeleton() {
  return (
    <div className="animate-pulse space-y-4 p-6">
      <div className="h-10 bg-muted rounded" />
      <div className="h-10 bg-muted rounded" />
      <div className="h-10 bg-muted rounded" />
      <div className="h-12 bg-muted rounded" />
    </div>
  )
}
```

2. Wrap the `<BookingForm>` (or the heavy client form component) with `<Suspense>`:

```tsx
import { Suspense } from 'react'

// In the page component:
<Suspense fallback={<FormSkeleton />}>
  <BookingForm ... />
</Suspense>
```

3. If the BookingForm is already wrapped in Suspense (check the current code first), skip adding a duplicate wrapper.

Apply to:
- `app/hk/book-a-trial/page.tsx`
- `app/hk/book-a-trial/free-assessment/page.tsx`
- `app/sg/book-a-trial/page.tsx`

Note: Do NOT add Suspense to SSR components. Only wrap client components (components with `'use client'` directive or imported dynamically).
</action>

- `grep "Suspense" app/sg/book-a-trial/page.tsx` returns Suspense usage
- The fallback prop is not empty (FormSkeleton or equivalent loading indicator)
