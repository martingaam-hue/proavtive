Create `.github/workflows/ci.yml` with this exact structure — one job named `ci`, four steps in fast-to-slow order so the cheap checks fail first. Keep logs readable (per Claude's discretion — D-02): one job is simpler to wire into branch protection than four parallel jobs. The speed-up from parallelism isn't worth the protection-config complexity for a solo dev project.

```yaml
name: ci
# Required status check for branch protection on `main`.
# Enforces D-02: typecheck + lint + build + gitleaks must all pass.
# Deferred (per D-03 / D-04): unit tests (Phase 2), axe a11y (Phase 7), Lighthouse CI (Phase 7).

on:
  pull_request:
    branches: [main]
  # Also run on pushes to main (belt-and-braces — a direct push would have already failed branch protection,
  # but this catches any admin-override bypass).
  push:
    branches: [main]

# Cancel superseded runs on the same ref to save minutes.
concurrency:
  group: ${{ github.workflow }}-${{ github.ref }}
  cancel-in-progress: true

jobs:
  ci:
    name: ci
    runs-on: ubuntu-latest
    timeout-minutes: 10
    permissions:
      contents: read
      # Required so gitleaks-action can post a redacted finding summary as a check-run annotation.
      checks: write
      # Allow the workflow to read secret-scanning alerts (D-07 context — not required, but useful).
      security-events: read

    steps:
      - name: Checkout
        uses: actions/checkout@v4
        with:
          # Fetch full history so gitleaks can scan the entire PR diff range.
          fetch-depth: 0

      # Corepack activates the exact pnpm version from package.json `packageManager`.
      - name: Enable Corepack
        run: corepack enable

      - name: Setup Node
        uses: actions/setup-node@v4
        with:
          node-version-file: '.nvmrc'
          cache: 'pnpm'

      - name: Install dependencies
        run: pnpm install --frozen-lockfile

      # --- Required check 1 (D-02): typecheck ---
      - name: Typecheck
        run: pnpm typecheck

      # --- Required check 2 (D-02): lint (ESLint + Prettier --check) ---
      - name: Lint
        run: pnpm lint
      - name: Format check
        run: pnpm format:check

      # --- Required check 3 (D-02): build ---
      - name: Build
        run: pnpm build
        env:
          # Sentry wizard adds SENTRY_AUTH_TOKEN to Vercel env in Plan 05; CI builds don't upload
          # source maps (no auth token here), which is fine — source map upload is a Vercel-build-only job.
          NEXT_TELEMETRY_DISABLED: '1'

      # --- Required check 4 (D-02 / D-07): gitleaks — layer 3 of the four-layer defense ---
      - name: Gitleaks
        uses: gitleaks/gitleaks-action@v2
        env:
          # GITLEAKS_LICENSE only required for org-scoped scanning on private repos — optional for Hobby solo use.
          GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}
        with:
          config-path: .gitleaks.toml
```

Key decisions baked into this file:
- **One job** (`ci`) — simpler for branch-protection required-check config. The four checks are four steps in the same job.
- **Fast-to-slow ordering** — typecheck (seconds) → lint → build (minutes) → gitleaks. If typecheck fails, we don't waste CI minutes on build.
- **`fetch-depth: 0`** — gitleaks needs the full diff range to scan PR history, not just the merge commit.
- **Corepack + `node-version-file: .nvmrc`** — CI uses the exact same Node + pnpm versions as local dev and Vercel (single source of truth from Plan 01).
- **Concurrency cancel-in-progress** — on rapid PR pushes, cancel the older run to save minutes.
- **`security-events: read` permission** — allows reading GH secret-scanning alerts if ever needed for audit; read-only, no write surface.
- **Format check included under "Lint"** per D-02's "ESLint + Prettier --check".
- **Do NOT echo any env var** — Actions auto-masks `secrets.*` but we don't add any `echo $SECRET` pattern at all (T-00-14 mitigation).

After creating the file, commit it on the same branch as Plan 03 (`chore/00-03-vercel-link`) OR on a new branch (`chore/00-04-ci`) — whichever is cleanest. Push the branch and open/update the PR. Verify the CI run triggers and goes green (scaffolded app from Plan 01 should pass typecheck, lint, build, gitleaks with zero findings).

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| developer machine → GitHub push API | `--no-verify` bypasses local hooks; branch protection + CI must catch |
| GitHub Actions runner → repo secrets | Workflow env must not echo secrets; runner environment is ephemeral but logs persist |
| GitHub admin → branch protection rules | Admin bypass (`enforce_admins: false`) is a conscious tradeoff for solo dev — document |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-00-14 | Information disclosure | secrets echoed in CI logs | mitigate | GitHub Actions auto-masks `secrets.*`. Additionally: no `echo` of env vars anywhere in the workflow (verified by acceptance criterion grep). No `set -x` / `bash -x`. Sentry + Vercel secrets are not referenced in this workflow at all (Plan 05 adds Sentry env only to Vercel's runtime env, not to CI). |
| T-00-15 | Elevation of privilege | admin bypass of branch protection | accept + document | `enforce_admins: false` is a conscious choice for solo dev (D-12 discretion). Documented in SUMMARY; flagged to revisit when collaborators join. Every bypass shows up in the repo audit log. |
| T-00-16 | Tampering | `--no-verify` local bypass getting to main | mitigate | Layer 3 CI gitleaks blocks the merge. Branch protection requires `ci` check green. No path to main without CI green. |
| T-00-17 | Information disclosure | private repo without GHAS missing push protection | accept + document | If GHAS is unavailable, lefthook (L1) + CI (L3) + D-08 negative test (L4) still catch secrets. Loss is redundancy only. Document in SUMMARY so future-Martin knows to revisit if upgrading. |
| T-00-18 | Tampering | CI workflow file edited in a PR to weaken checks | mitigate | Branch protection + `strict: true` (up-to-date requirement) means any PR editing `.github/workflows/ci.yml` still has to run under the OLD workflow's required checks before merging. Combined with `required_approving_review_count: 0` this is weaker than with reviewers — but for solo dev, the audit log is the compensating control. |
</threat_model>
